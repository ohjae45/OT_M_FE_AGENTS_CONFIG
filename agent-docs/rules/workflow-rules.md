# 작업 흐름 규칙

이 문서는 작업 전후에 따라야 할 공통 작업 원칙과 검증 기준을 정의한다.

## 작업 원칙

- 현재 프로젝트 요구사항을 최우선으로 따른다.
- 기존 코드 구조와 패턴을 먼저 확인한 뒤 작업한다.
- 새 폴더, 새 레이어, 새 추상화 추가 전 기존 구조와 충돌 여부를 먼저 검토한다.
- 미래를 위한 과한 추상화보다 현재 요구사항 충족을 우선한다.
- 아래 중 하나 이상 해당하면 구현 전에 구조안을 먼저 제시한다.
  - 새 레이어(디렉토리, 추상화 계층)를 도입하는 경우
  - 3개 이상 파일에 걸쳐 인터페이스가 변경되는 경우
  - 기존 공용 컴포넌트·훅의 시그니처가 바뀌는 경우
- 작업 전 기존 컴포넌트, 훅, 유틸 재사용 가능 여부를 먼저 확인한다.

## 구조 변경 승인 게이트

구조 변경은 구현 편의가 아니라 사용자 승인 후 수행하는 변경이다. 아래 조건 중 하나라도 해당하면 에이전트는 파일 생성, 수정, 이동, 삭제를 시작하기 전에 작업을 중단하고 구조안을 먼저 제시한다.

### 승인 게이트 발동 조건

- 새 디렉터리 또는 새 레이어(components 하위 전용 폴더, hooks, utils, types, styles 공유 파일 등)를 만드는 경우
- 컴포넌트, 훅, 유틸, 타입, 스타일 파일을 합쳐 3개 이상 새 파일로 분리하거나 추가하는 경우
- 기존 파일을 이동하거나 이름을 바꾸는 경우
- 3개 이상 파일의 props, 반환 타입, import 경로, 공용 인터페이스가 함께 바뀌는 경우
- 기존 공용 컴포넌트, 훅, 유틸, 스타일, 타입의 책임 경계를 바꾸는 리팩토링인 경우
- 여러 컴포넌트가 공유하는 SCSS Module, 타입 파일, 훅, 유틸을 새로 만드는 경우

### 구조안 필수 항목

구조안은 아래 항목을 빠짐없이 포함한다.

- 생성, 수정, 이동, 삭제할 파일 목록
- 각 파일의 단일 책임
- 컴포넌트, 훅, 유틸, 타입, 스타일로 나누는 기준
- 공유 스타일 또는 공유 타입을 만드는 경우 공유 범위와 이유
- 기존 구조와 비교해 달라지는 점
- 예상 검증 명령

### 승인 전 금지 행동

사용자의 명시적 승인 전에는 아래 작업을 하지 않는다.

- 새 파일 또는 새 디렉터리 생성
- 파일 이동 또는 이름 변경
- 공용 타입, 공용 훅, 공용 스타일 생성
- 기존 파일에서 대규모 JSX 추출
- 3개 이상 파일에 걸친 import 경로 변경

### 승인으로 인정하지 않는 표현

아래처럼 짧은 진행 알림은 구조안 제시나 승인으로 보지 않는다.

- "단계별 컴포넌트로 분리하겠습니다."
- "하위 컴포넌트로 나누겠습니다."
- "공유 스타일로 정리하겠습니다."

사용자가 "진행해", "이 구조로 해", "승인"처럼 명시적으로 답한 뒤에만 구조 변경을 시작한다.

## 외부 공개 게이트

본 조직의 코드, 문서, 결과물은 기본적으로 회사/팀 내부 바운더리 안에서만 유통한다. 에이전트는 아래 활동을 사용자에게 먼저 제안하지 않으며, 사용자가 명시적으로 외부 공개 의도를 밝힌 경우에만 수행한다.

### 기본 금지 제안 (사용자 명시 승인 전)

- **공개 패키지 저장소 publish** — npm registry (registry.npmjs.org), JSR, PyPI, crates.io 등 누구나 접근 가능한 패키지 저장소에 코드 publish 또는 그 절차 제안
- **공개 git 호스팅에 신규 repo 생성** — GitHub public repo, GitLab public project, Bitbucket public repo 등 외부 누구나 접근 가능한 형태로의 신규 저장소 생성
- **공개 외부 서비스에 코드 업로드** — CodeSandbox, CodePen, public Gist, JSFiddle, StackBlitz public 등 공개 코드 호스팅 서비스에 코드/스니펫 업로드
- **공개 CDN 에 자산 배포** — unpkg, jsDelivr 등 공개 CDN 으로의 자산 배포 제안
- **OSS 라이브러리화** — 사내 코드를 OSS 로 분리해 배포하는 절차 제안

### 허용 조건

위 활동은 사용자가 아래 중 하나에 해당하는 의사를 명시적으로 표현한 경우에만 진행한다.

- "OSS 로 공개", "외부 공유용", "공개 패키지로 배포", "퍼블릭 repo 로" 등 공개 의도 명시
- 회사 정책상 외부 공개가 승인된 결과물이라는 사실을 작업 지시에 포함
- 단순 짧은 진행 알림 ("배포해줘", "publish 해줘") 만으로는 공개 여부가 모호하므로, 에이전트가 먼저 "사내/공개" 를 묻고 명시 답변 확인 후 진행

### 사내 활용 시 권장 경로

같은 코드를 회사/팀 내부에서 재사용해야 할 경우 아래 옵션을 우선 검토한다.

- 모노레포 안 내부 워크스페이스 패키지 (`pnpm workspace`)
- 사내 GitHub org / GitLab group 의 private repo + 사내 npm proxy (예: Verdaccio, GitHub Packages private, JFrog Artifactory) 활용
- 단일 프로젝트 안 디렉토리 분리 + 잘 정의된 import 경계 유지 (대부분의 경우 충분)

### 새로 추가하는 항목/체크리스트 작성 시

작업 체크리스트나 TODO 문서에 "별도 repo 추출", "공개 패키지 publish" 같은 항목을 자동으로 추가하지 않는다. 외부 공개 수요가 실제 발생한 시점에만 사용자 확인 후 항목으로 인입한다.

## 리팩토링 불변조건

리팩토링은 기본적으로 기존 동작과 사용자 경험을 보존하는 작업이다. 파일 이동, 구조 분리, 컴포넌트/훅/유틸 추출을 수행할 때는 아래 항목을 의도 없이 변경하지 않는다.

- 화면의 시각적 크기, 간격, 색상, 배치
- API 호출 흐름과 데이터 가공 결과
- 상태 변경 타이밍과 이벤트 처리 흐름
- public import 경로를 제외한 외부 사용 계약

위 항목을 변경해야 하면 리팩토링 범위를 벗어난 별도 변경으로 보고, 변경 목적과 영향 범위를 사용자에게 먼저 공유한다.

## 변경 축 분리

한 번의 작업에서 변경 축을 섞지 않는다. 에이전트는 작업 시작 전 주요 변경 축을 구분하고, 현재 작업의 축을 벗어나는 변경을 발견하면 작업을 멈추고 사용자에게 공유한다.

- 구조 변경
- 책임 분리
- 네이밍 정리
- 스타일 값 변경
- 동작 변경

여러 축이 필요한 작업은 순서를 나누어 진행하고, 각 단계마다 보존할 항목과 검증 방법을 밝힌다.

## 구조 변경 문서화 규칙

구조 변경 승인 게이트에 해당하는 작업을 실제로 수행한 경우, 작업 후 관련 문서를 함께 갱신한다.

- 신규 도메인, 페이지, 라우트, feature 모듈 추가
- 폴더 구조, 파일 위치, import boundary, ownership 변경
- 공통 hook, service, API client, store, util 등 재사용 구조 추가 또는 변경
- 프로젝트 전용 구현 규칙, 네이밍 규칙, 예외 규칙 추가

위 항목에 해당하는 프로젝트 구조 또는 도메인 변경 내용은 반드시 해당 프로젝트의 `AGENTS.md`에 갱신한다.
단순 버그 수정, 스타일 수정, 문구 수정, 컴포넌트 내부 구현 변경만 있는 경우에는 문서를 갱신하지 않는다.
불확실한 경우에는 임의로 문서를 수정하지 말고 사용자에게 먼저 확인한다.

## 리팩토링 원칙

- 한 파일 내 컴포넌트를 여러 파일로 분리하기 전, 같은 파일 또는 같은 디렉터리의 기존 컴포넌트 구조를 먼저 파악하고 기준 형식으로 삼는다.
- 기준 형식 확인 항목은 파일 구성, import 방식, props 정의 위치 등이다.
- 분리가 여러 단계에 걸치더라도 초기에 확인한 기존 구조를 유지한다.
- 중간에 구조가 달라지는 경우 작업을 멈추고 사용자에게 공유한다.

## 검증 규칙

- 작업 후 아래 명령어를 우선 확인한다.
  - `pnpm lint`
  - `pnpm typecheck`
- PR, 배포, 공유 전에는 필요 시 `pnpm build`까지 확인한다.
- 검증 명령이 실패하면 수정하거나, 실패 사유와 미확인 범위를 사용자에게 공유한다.
