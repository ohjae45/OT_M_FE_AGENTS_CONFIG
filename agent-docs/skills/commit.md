# /commit

## Description
현재 staged/unstaged 변경사항을 분석하여 커밋 메시지를 생성합니다.
브랜치명에서 JIRA 이슈 번호를 자동으로 추출합니다.

## 브랜치 네이밍 규칙
```
{키워드}/{JIRA Key}
```
예: `feat/ONT-49`, `fix/ONT-102`, `conf/ONT-120`

## 커밋 메시지 형식
```
{키워드}: 커밋 내용
```
예: `feat: 로그인 기능 추가`, `fix: 토큰 만료 시 자동 로그아웃 문제 해결`

## 키워드 정의
| 키워드 | 설명 |
|--------|------|
| `feat` | 새로운 기능 추가 (화면 최초 구현 포함) |
| `improve` | 기존 기능 개선 및 수정 |
| `fix` | 버그 및 에러 수정 |
| `style` | 디자인·UI 변경 (기능 영향 없음, 이미 존재하는 화면의 CSS 수정) |
| `refactor` | 코드 리팩터링 (기능 변화 없음) |
| `docs` | 문서·주석 추가 및 수정 (README, 위키 등) |
| `conf` | 설정 파일 수정 (eslint, vite, env, tsconfig 등) |
| `chore` | 유지보수·패키지 관리 (PR 템플릿, GitHub Actions, 의존성 정리 등) |
| `deploy` | 배포 관련 변경 |
| `test` | 테스트 코드 추가·수정 |

### style vs feat 구분
| 상황 | 타입 |
|------|------|
| 화면/페이지 최초 구현 | `feat` |
| 이미 존재하는 페이지의 디자인만 수정 | `style` |
| 이미 존재하는 페이지에 기능 추가 | `feat` |
| 퍼블리싱만 완성하고 기능은 미구현 | `style` |

## Steps

1. `git status`와 `git diff HEAD`로 변경사항 파악
2. 현재 브랜치명에서 JIRA 이슈 번호 추출 (예: `feat/ONT-123` → `ONT-123`)
3. 변경 내용에 맞는 키워드 선택
4. 커밋 메시지 생성:
   ```
   {키워드}: {한국어 요약}
   ```
   이슈 번호가 있으면:
   ```
   {키워드}: {한국어 요약}

   {이슈번호}
   ```
5. 생성된 메시지를 제안하고, 사용자 확인 후 `git commit -m "..."` 실행

## Rules
- 소문자 키워드로 시작
- 제목은 50자 이내
- 한국어로 작성 (영어 고유명사 제외)
- 명령형 어조 사용 ("추가", "수정", "제거")
- `git add`는 실행하지 않음 — 사용자가 직접 스테이징
- 커밋 실행 전 반드시 메시지 확인 요청
