# /pr

## Description

작업한 내용을 바탕으로 PR 설명을 작성합니다.
"PR 작성", "PR 써줘", "PR 올려줘", "풀리퀘 작성", "작업 내용으로 PR" 등의 키워드로 트리거됩니다.

## Trigger Keywords

- PR 작성 / PR 써줘 / PR 올려줘
- 풀리퀘 / 풀리퀘스트 작성
- 작업한 내용을 바탕으로 PR
- PR 내용 만들어줘

## PR 제목 형식

```
[키워드]: PR 제목
```

- 키워드는 **대문자**로 시작
- 예: `[Feat]: 로그인 페이지 구현`, `[Fix]: 토큰 만료 오류 수정`, `[Conf]: vite 환경변수 설정 변경`

## Steps

1. 현재 브랜치명에서 이슈 번호 추출

   ```
   git branch --show-current
   ```

   예: `feat/ONT-241` → `ONT-241`

2. 변경사항 파악

   ```
   git log dev...HEAD --oneline
   git diff dev...HEAD --stat
   git diff dev...HEAD
   ```

3. `.github/pull_request_template.md` 파일을 읽어 구조와 섹션을 파악한 뒤, 해당 형식에 맞춰 내용을 채움

4. 작성한 PR 본문을 **마크다운 코드 블록(`markdown ... `)으로 감싸서 출력** — 사용자가 블록 복사 버튼으로 한 번에 복사할 수 있도록

## 작성 규칙

- **마크다운 코드 블록으로 출력**: 파일 저장 및 `git` 명령어 제출 금지
- **이슈 번호**: 브랜치명에서 자동 추출, 없으면 사용자에게 질문
- **작업 개요**: 어떤 문제가 발생했는지 상황 중심으로 서술
- **작업 내용**: 항목마다 백틱으로 파일명·함수명 강조, 세부 내용은 들여쓰기 불릿으로 작성
- **추가 작업 섹션**: 부수적 변경이 없으면 섹션 통째로 생략
- **문체**: 간결하고 직접적인 서술체. "~함", "~됨" 또는 명사형 종결

## 예시 출력 톤

나쁜 예: "기능을 개선하여 성능을 향상시켰어요"
좋은 예:

- `ComponentName.tsx` 기능명 추가
  - 세부 변경사항 1
  - 세부 변경사항 2
