# /skai-pr

## 설명

작업한 내용을 바탕으로 PR 설명을 작성합니다.
사용자가 "PR 작성", "PR 써줘", "PR 올려줘", "풀리퀘 작성", "풀리퀘스트 작성", "작업 내용으로 PR", "PR 내용 만들어줘"처럼 요청할 때 사용합니다.

## 중단 조건

- `.github/pull_request_template.md`가 없으면 즉시 중단한다.
- 사용자에게 템플릿 추가 또는 템플릿 내용 제공을 요청한다.
- `dev` 대비 변경사항이 없으면 즉시 중단하고 사용자에게 알린다.

## 금지 사항

- `.github/pull_request_template.md`가 없으면 PR 제목/본문을 작성하지 않는다.
- 임의 형식, 대체 형식, 추정 형식으로 PR을 작성하지 않는다.
- 파일 저장 및 `git` 명령어 제출을 하지 않는다.

## 진행 단계

1. `입력 기준`에 따라 이슈 번호 확인
2. `dev` 대비 변경사항 파악

   base 브랜치는 `dev`로 고정한다.

   ```
   git log dev...HEAD --oneline
   git diff dev...HEAD --stat
   git diff dev...HEAD
   ```

3. `.github/pull_request_template.md` 파일을 읽어 구조와 섹션을 파악한 뒤, 해당 형식에 맞춰 내용을 채움

4. `출력 형식`에 맞춰 작성한 PR 본문을 출력

## 입력 기준

- 이슈 번호는 브랜치명에서 자동 추출한다.
- 브랜치명에서 이슈 번호를 찾을 수 없으면 사용자에게 질문한다.
- 이슈 번호 추출 시 `git branch --show-current`를 사용한다.

## 출력 형식

- 작성한 PR 본문은 마크다운 코드 블록으로 감싸서 출력한다.

## PR 제목 형식

```
[키워드]: PR 제목
```

- 키워드 정의와 구분 기준은 `agent-docs/rules/git-keyword-rules.md`를 따른다
- PR 제목에는 공통 키워드를 PascalCase로 변환해 사용한다
- 공통 키워드 목록에 없는 임의 키워드는 사용하지 않는다
- 예: `feat` → `[Feat]`, `fix` → `[Fix]`, `conf` → `[Conf]`
- 예: `[Feat]: 로그인 페이지 구현`, `[Fix]: 토큰 만료 오류 수정`, `[Conf]: vite 환경변수 설정 변경`

## 작업 원칙

- **작업 개요**: 문제 상황 중심으로 작성한다. 단일 내용은 완결된 문장형 서술체로, 2개 이상은 리스트로 작성한다.
- **작업 내용**: 작성자가 아닌 팀원도 변경 의도와 핵심 구현을 이해할 수 있게 간결하게 작성한다.
- **코드 표현**: 컴포넌트명, 파일명, 함수명 등 코드 관련 이름은 백틱으로 표시한다.
- **작업 내용 형식**: 작업은 리스트로 작성하고, 세부 내용은 한 단계까지만 중첩한다.
- **주요/추가 작업 구분**: 부수 작업이 있을 때만 `주요 작업`과 `추가 작업`으로 나눈다.
- **참고사항**: 단일 내용은 완결된 문장형 서술체로, 2개 이상은 리스트로 작성한다.
- **작업 내용 문체**: 간결한 리스트형 표현을 사용하고, "~함", "~됨" 또는 명사형으로 끝낸다.

## 섹션 작성 예시

### 작업 개요

단일 내용이면 인용구 없이 완결된 문장형 서술체로 작성한다.

```
로그인 폼 입력값 검증 기준이 없어 잘못된 사용자 정보가 제출될 가능성이 있었습니다.
```

서로 구분되는 내용이 2개 이상이면 리스트로 작성한다.

```
- 로그인 폼 입력값 검증 기준이 누락되어 있었습니다.
- 인증 실패 시 사용자에게 표시되는 피드백이 부족했습니다.
```

### 작업 내용

부수 작업이 없으면 `작업 내용` 아래에 바로 리스트로 작성한다.

```
- 사용자 아이디, 비밀번호의 유효성 검사 로직 생성
- `LoginForm` 제출 전 입력값을 검증해 잘못된 로그인 요청 차단
  - 빈 값, 최소 길이, 형식 오류 케이스 처리
- `validateLoginInput` 함수 추가
```

현재 브랜치의 큰 맥락과 거리가 있는 부수 작업이 있으면 `주요 작업`, `추가 작업`으로 나누어 작성한다.

```
#### 주요 작업

- 사용자 아이디, 비밀번호의 유효성 검사 로직 생성
- `LoginForm` 제출 전 입력값을 검증해 잘못된 로그인 요청 차단
  - 빈 값, 최소 길이, 형식 오류 케이스 처리

#### 추가 작업

- `Button` 컴포넌트 import 경로 정리
- 사용하지 않는 상수 제거
```

### 참고사항

단일 내용이면 인용구 없이 완결된 문장형 서술체로 작성한다.

```
기존 로그인 API 요청 형식은 유지했습니다.
```

서로 구분되는 내용이 2개 이상이면 리스트로 작성한다.

```
- 기존 로그인 API 요청 형식은 유지했습니다.
- 유효성 검사 문구는 추후 기획 확정 후 조정이 필요합니다.
```
