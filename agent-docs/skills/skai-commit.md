# /skai-commit

## 설명

커밋 메시지를 생성하고 사용자 확인 후 커밋을 실행합니다.
사용자가 "커밋 메시지 작성", "커밋해줘", "커밋 만들어줘", "커밋 메시지 추천"처럼 요청할 때 사용합니다.

## 중단 조건

- staged 변경사항이 없으면 즉시 중단하고 사용자에게 스테이징을 요청한다.

## 금지 사항

- `git add`는 실행하지 않는다.
- unstaged 변경사항은 커밋 메시지에 반영하지 않는다.
- 사용자가 승인하지 않은 경우 임의로 `git commit -m "..."`을 실행하지 않는다.

## 진행 단계

1. `git status --short`로 staged/unstaged 변경사항을 구분
2. staged 변경사항이 없으면 중단 조건에 따라 중단
3. `git diff --cached`로 커밋 대상 변경사항만 분석
4. unstaged 변경사항이 있으면 사용자에게 "이번 커밋 대상 아님"으로 별도 안내
5. staged 변경 내용에 맞는 키워드 선택
6. `커밋 메시지 형식`에 맞춰 커밋 메시지 생성
7. 생성된 커밋 메시지를 먼저 사용자에게 제안하고, 사용자가 승인한 경우에만 `git commit -m "..."` 실행

## 커밋 메시지 형식

```
{키워드}: 커밋 내용
```

예: `feat: 로그인 기능 추가`, `fix: 토큰 만료 시 자동 로그아웃 문제 해결`

## 커밋 내용 작성 기준

- 커밋 내용에는 staged 변경사항의 대상 범위를 드러낸다.
- 문서 변경은 어떤 문서나 규칙을 수정했는지 포함한다.
- 설정 변경은 어떤 설정 영역을 수정했는지 포함한다.
- 기능 변경은 어떤 화면, 도메인, 흐름을 수정했는지 포함한다.
- 변경 대상 없이 작업 내용만 적어 의미가 넓어지는 제목은 피한다.

```txt
Good: docs: 프론트 공통 규칙에 path alias 기준 추가
Bad: docs: path alias 기준 추가
```

## 키워드 기준

키워드 정의와 구분 기준은 `agent-docs/rules/git-keyword-rules.md`를 따른다.
커밋 메시지에는 공통 키워드를 소문자로 사용한다.

## 작업 원칙

- 커밋 메시지는 staged 변경사항만 기준으로 작성
- 소문자 키워드로 시작
- 제목은 50자 이내
- 한국어로 작성 (영어 고유명사 제외)
- 명령형 어조 사용 ("추가", "수정", "제거")
- 커밋 실행 전 반드시 메시지 확인 요청
