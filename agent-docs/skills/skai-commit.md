# /skai-commit

## 설명

사용자가 커밋과 관련된 어떤 요청이든 했을 때 반드시 이 스킬을 사용한다. 기본 git 워크플로우(직접 `git add` + `git commit`)를 사용하지 말고 이 스킬의 그룹핑·승인·커밋 흐름을 따른다. 트리거 표현: "커밋해줘", "커밋 만들어줘", "커밋 메시지 작성", "커밋 메시지 추천", "정리해서 커밋", "이거 커밋", "PR 전에 커밋". 워크트리의 변경 사항을 작업 단위로 그룹핑하고 사용자 승인을 받아 그룹별로 커밋한다. 미리 스테이징이 분리되어 있지 않아도 메인 에이전트가 변경을 분석하고 분리 안을 제안한다.

## 중단 조건

- 워크트리에 변경 사항(staged/unstaged/untracked)이 전혀 없으면 즉시 중단하고 사용자에게 알린다.
- 그룹핑 안에 사용자가 승인하지 않으면 어떤 변경도 commit하지 않는다.

## 금지 사항

- 사용자가 그룹핑 안과 커밋 메시지를 승인하기 전에는 `git add`도 실행하지 않는다.
- 사용자가 메시지를 승인하지 않은 상태로 `git commit -m "..."`을 실행하지 않는다.
- hunk 단위 분리(`git add -p`)는 수행하지 않는다. 분리 단위는 항상 파일이다.
- 같은 파일을 두 그룹에 동시에 넣지 않는다. 한 파일은 한 그룹에만 속한다.
- 환경/비밀 파일(`.env*`, `*.key`, `credentials.*`, `*.pem` 등)이 변경 목록에 보이면 그룹에 넣지 말고 사용자에게 먼저 확인한다.

## 진행 단계

1. `git status --short`로 변경 사항을 확인한다(staged, unstaged, untracked 모두 포함).
2. 변경 사항이 없으면 중단 조건에 따라 중단한다.
3. 이미 staged 상태인 파일이 있으면 사용자에게 한 줄로 알리고, `git reset HEAD -- <files>`로 풀어 워크트리 상태에서 다시 정리한다. 사용자가 기존 staged 상태를 보존하고 싶다고 명시하면 그 부분은 손대지 않고 그룹핑에서 제외한다.
4. `git diff`, `git diff --stat`, untracked 파일의 경우 파일 내용을 직접 읽어 변경의 의도(intent)를 파악한다.
5. 변경을 작업 단위로 그룹핑한다. 그룹핑 기준은 아래 "그룹핑 기준" 섹션을 따른다.
6. 그룹 수에 따라 진행한다.
   - 그룹이 1개이면 단일 커밋 메시지 후보를 작성한다(기존 단일 커밋 흐름).
   - 그룹이 2개 이상이면 각 그룹별로 (대상 파일 + 메시지 후보)를 정리해 사용자에게 일괄 제안한다.
7. 한 파일에 여러 의도가 섞인 것으로 의심되면 그룹핑을 강행하지 말고 사용자에게 "이 파일에 여러 의도가 섞여 보이는데 어떻게 처리할까요?"라고 먼저 묻는다.
8. 그룹핑 안과 커밋 메시지를 "출력 형식"에 따라 제안하고 사용자 승인을 받는다.
9. 승인된 그룹을 순서대로 처리한다. 그룹마다 `git add <files>` → `git commit -m "..."`를 순차 실행한다.
10. 모든 그룹 처리 후 `git status`와 `git log --oneline -N`(N = 그룹 수)으로 결과를 사용자에게 공유한다.

## 그룹핑 기준

- 작업 단위는 한 가지 의도(intent)를 가진 변경 묶음을 말한다. 같은 도메인을 건드렸더라도 의도가 다르면 분리한다.
- 다음 신호를 같은 그룹으로 묶는 근거로 본다.
  - 같은 기능, 같은 화면, 같은 도메인 경로
  - 같은 규칙 문서/스킬/에이전트 정의에 대한 일관된 수정
  - 변경된 코드와 그에 대응하는 테스트
  - 동일 변경 축(스타일 값, 동작, 구조 변경 등) 안의 변경
- 다음 신호는 다른 그룹으로 분리하는 근거로 본다.
  - 무관한 리팩토링과 새 기능이 함께 들어 있음
  - 문서 갱신만 다른 영역을 다루고 있음
  - 설정 파일 변경(`conf`)과 기능 변경(`feat`/`improve`)이 함께 있음
  - 한 commit의 제목으로 모든 변경을 정확히 요약하기 어려움
- 파일 단위로만 분리한다. 한 파일이 여러 의도를 동시에 담은 것으로 보이면 사용자에게 처리 방식을 먼저 확인한다.

## 출력 형식

그룹이 2개 이상일 때 다음 형식으로 제안한다.

```
## 커밋 그룹 제안

### 그룹 1: {짧은 의도 요약}
- 대상 파일:
  - {path}
  - {path}
- 메시지 후보:
  - A) {키워드}: {제목 A}
  - B) {키워드}: {제목 B}
- 선택: A — {선택 사유 한 줄}
- 본문 요약(선택): {핵심만}

### 그룹 2: {짧은 의도 요약}
...

### 진행 순서
1. 그룹 1
2. 그룹 2

### 사용자 확인 필요
- 위 그룹핑과 메시지로 진행해도 될지 명시적 승인
```

그룹이 1개이면 그룹 헤더 없이 메시지 후보만 제안한다.

## 커밋 메시지 형식

기본 형식은 다음과 같다.

```
{키워드}: 커밋 내용
```

예: `feat: 로그인 기능 추가`, `fix: 토큰 만료 시 자동 로그아웃 문제 해결`

### Jira 티켓 번호가 함께 전달된 경우

사용자가 입력에 Jira 티켓 번호를 함께 전달하면 키워드 뒤에 `(Jira-{번호})` 형식으로 붙인다.

```
{키워드}(Jira-{번호}): 커밋 내용
```

예: `feat(Jira-1234): 로그인 기능 추가`, `fix(Jira-5678): 토큰 만료 시 자동 로그아웃 문제 해결`

티켓 번호 인식 기준:

- `Jira-1234`, `JIRA-1234`, `jira 1234`, `티켓 1234`, `#1234`처럼 명시적으로 전달된 숫자
- 사용자 메시지에 `PROJ-1234` 형태의 표준 Jira 키가 포함된 경우 그대로 사용 (`feat(PROJ-1234): ...`)

여러 그룹을 만들 때 사용자가 한 티켓 번호만 전달했다면, 모든 그룹의 메시지에 같은 번호를 붙인다. 그룹별로 다른 티켓 번호를 명시했다면 각 그룹에 해당 번호를 매핑한다. 티켓 번호가 어느 그룹에 해당하는지 불명확하면 그룹핑 안과 함께 사용자에게 확인을 받는다.

티켓 번호가 입력에 없으면 기본 형식(`{키워드}: ...`)을 사용한다. 추측하거나 자동 생성하지 않는다.

## 커밋 내용 작성 기준

- 커밋 내용에는 해당 그룹 변경 사항의 대상 범위를 드러낸다.
- 문서 변경은 어떤 문서나 규칙을 수정했는지 포함한다.
- 설정 변경은 어떤 설정 영역을 수정했는지 포함한다.
- 기능 변경은 어떤 화면, 도메인, 흐름을 수정했는지 포함한다.
- 변경 대상 없이 작업 내용만 적어 의미가 넓어지는 제목은 피한다.
- 제목은 `작업 대상 + 변경 기준`이 함께 드러나게 작성한다.
- 단순히 파일명이나 넓은 레이어만 적지 말고 그룹 변경의 공통 의도를 포함한다.
- `정리`, `수정`, `개선`, `반영`처럼 넓은 표현은 변경 기준이 같이 있을 때만 사용한다.

```txt
Good: docs: 프론트 공통 규칙에 path alias 기준 추가
Bad: docs: path alias 기준 추가

Good: refactor: graphName과 graphPath 변수 의미 정리
Bad: refactor: 그래프 변수명 정리

Good: refactor: API 요청 함수 반환 타입 명시
Bad: refactor: API 타입 정리
```

## 커밋 메시지 품질 점검

그룹마다 메시지를 제안하기 전에 후보를 2개 이상 만들고 더 구체적인 제목을 선택한다.

- 제목만 보고 어떤 도메인/레이어가 바뀌었는지 알 수 있는가?
- 제목만 보고 어떤 기준으로 바뀌었는지 알 수 있는가?
- `정리`, `수정`, `개선`, `반영` 같은 표현만 남고 구체 대상이 빠지지 않았는가?
- 그룹 변경 일부만 설명하거나, 반대로 변경 범위보다 넓게 말하지 않았는가?

## 키워드 기준

- 키워드 정의와 구분 기준은 `agent-docs/rules/git-keyword-rules.md`를 따른다.
- 커밋 메시지에는 공통 키워드를 소문자로 사용한다.

## 작업 원칙

- 커밋 메시지는 사용자가 승인한 그룹의 변경 사항만 기준으로 작성한다.
- 소문자 키워드로 시작한다.
- 제목은 50자 이내로 작성한다.
- 본문은 선택이며, 작성한다면 핵심 한 줄에서 최대 3줄로 유지한다. 변경 사항 나열은 피한다.
- 한국어로 작성한다(영어 고유명사 제외).
- 명령형 어조를 사용한다("추가", "수정", "제거").
- 커밋 실행 전 반드시 그룹핑 안과 메시지 모두 사용자 확인을 받는다.
- 그룹 처리 도중 실패가 발생하면 즉시 중단하고 현재까지의 결과를 사용자에게 보고한다.
